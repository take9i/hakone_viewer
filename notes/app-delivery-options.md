# Cesium CAD ビューアの提供方法 — 検討まとめ

本資料では、Cesium を CAD データ可視化のベースとして構築した Web アプリ（Web サービス）の**提供方法**について、既存プラットフォームの検討・自前構築との比較・自前の場合の定番構成をまとめる。対象は課金・プラン制限・CAD データ管理を伴う提供を想定している。

---

## 1. 提供方法の選択肢

### 1.1 既存プラットフォームに載せる


| プラットフォーム                         | 配布の形                                   | 課金・プラン                                           | CAD データ管理                                                             | 備考                                                        |
| -------------------------------- | -------------------------------------- | ------------------------------------------------ | --------------------------------------------------------------------- | --------------------------------------------------------- |
| **Salesforce（AppExchange）**      | マネージドパッケージ（LWC + iframe で自前アプリ URL 表示） | リスティングの価格プラン、Commerce 連携など                       | ContentVersion（バイナリ）＋カスタムオブジェクト（メタデータ）。大きい CAD は S3 等＋参照のみ SF に保存が現実的 | セキュリティレビュー必須。CRUD/FLS 準拠・外部連携の開示が必要。既存 Salesforce 顧客向けに強い |
| **kintone**                      | プラグイン／ガジェットで iframe に自前アプリ URL 表示      | 公式のアプリ課金はなし。Ribbit 等のプラグインストアで Stripe 等、または自社で契約 | 添付ファイル（1 ファイル 10MB 目安）。大きい CAD は S3 等＋レコードで URL・メタデータ                 | 公式の統一審査はない。中小で kintone 利用が多いため、既存 kintone 顧客向けに有効         |
| **Microsoft（AppSource + Azure）** | SaaS オファーで配布。実体は Azure 上に自前構築          | 商用マーケットプレースでサブスク価格設定可能                           | Azure Blob + DB。自前で実装                                                 | 既存 Microsoft 契約企業向け                                       |


### 1.2 自前で Web アプリを構築して提供

- 自社サイトまたはランディングページから申し込み・ログイン・利用まで完結させる。
- 課金・プラン制限・CAD データ管理はすべて自前（または後述の定番サービス群）で実装する。
- 必要に応じて、後から kintone 連携や Salesforce 連携を「オプション」として追加できる。

## 2. 既存プラットフォーム vs 自前構築 — 比較


| 観点             | 既存プラットフォーム（Salesforce / kintone 等）                                          | 自前構築                                     |
| -------------- | --------------------------------------------------------------------------- | ---------------------------------------- |
| **課金・請求**      | マーケットの価格設定やプラットフォームの請求と連携可能                                                 | Stripe 等で自前実装。プラン設計・価格は自由                |
| **プランによる機能制限** | プラットフォームのライセンス・オブジェクトと連携して実装                                                | 自前 DB で「ユーザ／組織 ⇔ プラン」を管理し、API で制限        |
| **CAD データ管理**  | ContentVersion / kintone 添付 / Azure Blob 等を利用。既存顧客のデータがそのプラットフォームにあると統合しやすい | S3 + 自前 DB で設計の自由度が高い                    |
| **配布・発見**      | AppExchange / プラグインストア等で既存ユーザにリーチしやすい                                       | 自社サイト・SEO・広告・営業に依存。マーケットのトラフィックは使えない     |
| **審査・制約**      | セキュリティレビュー（SF）やストア規定に準拠が必要                                                  | アプリのプラットフォーム審査はない。決済は Stripe 等の規約に従う     |
| **初期開発**       | パッケージ／プラグイン開発とプラットフォーム仕様への適合に工数                                             | 認証・API・ストレージ・フロントを一通り構築する工数              |
| **運用・保守**      | プラットフォームのアップデート・仕様変更の影響を受ける                                                 | インフラ・アプリすべて自前で監視・障害対応が必要                 |
| **コスト構造**      | パートナー費用・マーケット手数料・利用組織のライセンス料                                                | インフラ・決済手数料のみ。プラットフォーム利用料はなし              |
| **ターゲット**      | すでにそのプラットフォームを導入している組織に「追加アプリ」として売りやすい                                      | どの企業にも売れるが、営業・オンボーディング・サポートを自前で用意する必要がある |


**「課金したい」という動機だけでは、Salesforce 等を使う（使ってもらう）のは過剰になることが多い。** ターゲットがそのプラットフォームをすでに使っているか、データやワークフローをそこでまとめたいかどうかで、プラットフォーム利用の妥当性が決まる。

## 3. 自前構築の場合の定番の方法

「Web アプリを課金したい」というユースケースは多く、以下のような構成が**定番**として使われている。

### 3.1 技術スタックの例


| 役割           | 定番の選択肢                                                     | 備考                                                                                                    |
| ------------ | ---------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **課金・サブスク**  | **Stripe**                                                 | サブスク・プラン・メータリング・Customer Portal。日本語対応・事例が豊富                                                           |
| **認証**       | **Supabase Auth** / **Clerk** / **Auth0**                  | メール・パスワード、OAuth。組織単位でプラン管理する場合は「組織＋メンバー」を自前 DB で持つ                                                    |
| **DB**       | **Supabase**（PostgreSQL）/ **Firebase** / **PlanetScale** 等 | ユーザ・組織・プラン・メタデータを保存。Supabase は Auth＋DB＋Storage をまとめて利用可能                                              |
| **ファイル保存**   | **Supabase Storage** / **AWS S3** / **Cloudflare R2**      | CAD 等の大容量は S3 / R2 が一般的                                                                               |
| **フロント・API** | **Next.js**（App Router）/ **React + Vite** 等                | API Routes や Server Actions で Stripe Webhook・課金状態の更新。Cesium は既存の Vite+React のままでも、Next.js の一部に埋め込んでも可 |
| **ホスティング**   | **Vercel** / **AWS** / **Cloud Run** 等                     | Next.js なら Vercel が相性が良い                                                                              |


### 3.2 課金フローのイメージ

1. ユーザがプラン選択 → **Stripe Checkout** へ遷移（または Stripe API でサブスク作成）。
2. 決済完了 → **Stripe Webhook**（`checkout.session.completed` 等）で自前サーバが受け取り、DB の「このユーザ／組織は有料プラン」を更新。
3. アプリ側では **Feature Gate**：API や画面でプランを参照し、機能制限（例: 有料のみエクスポート可）を切り替える。

### 3.3 営業面でのデメリット（自前販売の場合）

- **認知・発見**: マーケットプレイスのような「検索して見つかる」導線がない。SEO・広告・展示会・紹介・営業で集客する必要がある。
- **信頼**: 「プラットフォーム掲載」「審査済み」のような背押しがない。自社ブランド・事例・デモで信頼を積む必要がある。
- **購入までの導線**: 見積・契約・請求を自前で用意する必要があり、初回の申し込みステップは増えがち。

**業界・用途がはっきりしている場合**（例: 中小建設会社向け提案用 CAD ビューア）は、マーケットをあまり見ない顧客が多いため、自前販売の営業的デメリットは相対的に小さく、業界向けの集客・営業で補う形が現実的である。

## 4. ターゲットに応じたおすすめ（中小建設会社・提案用を想定した場合）

- **第一のすすめ**: **自前 Web アプリを主軸**にする。  
  - 中小では Salesforce は過剰になりやすく、提案用に特化した体験（アップロード → 地図・日影表示 → 共有リンク）を自前で設計しやすい。  
  - 月額数千円〜のサブスクで提供しやすく、Stripe + S3 + DB で課金・プラン制限・CAD 管理を賄える。
- **第二のすすめ**: **kintone を「もう一つの入口」**にする。  
  - 既に kintone で案件・見積を管理している顧客向けに、プラグイン／ガジェットで同じ Cesium ビューアを iframe 表示する。  
  - 課金は「kintone 連携オプション」として自社サイトで契約するか、プラグインストアの課金を利用する。
- **Salesforce**: 中小建設会社がすでに Salesforce を導入しているケースは限定的なため、まずは自前 + kintone で検討し、大口・エンタープライズ需要が出た段階で AppExchange を検討する形が現実的。

## 5. 補足

- **DB をユーザに使わせる部分**（メタデータ・物件情報の登録・検索 UI）の工数軽減は、別タイミングで検討する（Airtable を UI として使う、Refine で CRUD 自動生成する、など）。
- 影表示の技術的制約・対応方針については、[Cesium 影表示の現状・原因・対応方法](cesium-shadow-issues.md) を参照。
