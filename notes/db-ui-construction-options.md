# DB 部分（メタデータ・物件の登録・検索 UI）の構築方法 — 検討まとめ

本資料では、**メタデータや物件の情報の登録・検索**をユーザーに使わせる部分の UI を、工数を抑えて構築するための案と、各案での **CAD アップロード・メタデータ付与・検索結果の表示** の実現方法、および **MVP としての推奨構成・認証** をまとめる。

---

## 1. 目的と選択肢の概要

- **目的**: 登録・検索 UI の構築コストを抑えつつ、CAD とメタデータの登録・検索・Cesium ビューアでの表示までを実現する。
- **主な選択肢**: (1) Airtable を UI にする、(2) kintone を UI にする、(3) Refine で CRUD を自動生成、(4) 必要最小限の自前フォーム＋一覧。

| 案 | 登録・検索 UI の工数 | 備考 |
|----|----------------------|------|
| **Airtable** | ほぼゼロ（API 連携のみ） | 料金・制限あり。認証・課金は自前。 |
| **kintone** | ほぼゼロ（kintone 側で設計） | kintone 利用者向けに限定。 |
| **Refine + Supabase** | 小（スキーマ＋リソース設定） | 自前 DB。管理画面寄りの UI。 |
| **最小限の自前** | 中（1 フォーム＋1 テーブル） | 自前 DB。UI をシンプルに保てる。 |

---

## 2. 各案での「CAD アップロード」と「メタデータ付与」

### 2.1 Airtable を UI にする場合

| 項目 | 実現方法 |
|------|----------|
| **CAD のアップロード** | Airtable の **添付ファイル（Attachment）フィールド** に、ユーザーがブラウザから直接アップロードする。1 レコード = 1 物件とし、そのレコードに CAD ファイルを添付する。**大きな CAD**（例: 10MB 超）は、自前で S3 等にアップロードする画面を 1 つ用意し、Airtable には「CAD の URL」だけを格納する運用も可能。 |
| **メタデータの付与** | **同じレコードの他のフィールド**（テキスト・数値・選択肢・日付など）に、敷地形状・道路幅・家族構成・趣味・緯度経度などを入力する。登録時は「ファイルを添付した行」のセルにメタデータを打ち込む。 |

### 2.2 kintone を UI にする場合

| 項目 | 実現方法 |
|------|----------|
| **CAD のアップロード** | kintone の **添付ファイルフィールド** に、ユーザーがレコード詳細画面などからファイルを添付する。**大きい CAD** は S3 等に自前アップロードし、kintable には「ファイル URL」だけを格納する方法も取れる。 |
| **メタデータの付与** | **同じレコードの他のフィールド** に入力する。レコード追加／編集画面でファイルを添付しつつ、同じ画面でメタデータを入力する。 |

### 2.3 Refine + Supabase の場合

| 項目 | 実現方法 |
|------|----------|
| **CAD のアップロード** | Refine の作成／編集画面にファイル入力を追加する。送信時に **Supabase Storage**（または S3）にアップロードし、返ってきた **URL を DB の「CAD ファイル URL」カラム** に保存する。 |
| **メタデータの付与** | **同じ作成／編集画面** に、敷地形状・道路・法規・家族構成・趣味・緯度経度などの項目を並べる。1 画面でファイルとメタデータをまとめて登録・更新する。 |

### 2.4 最小限の自前フォーム＋一覧の場合

| 項目 | 実現方法 |
|------|----------|
| **CAD のアップロード** | 1 つの登録フォームにファイル入力欄を用意し、送信時に **S3 または Supabase Storage** にアップロードする。保存後に得た URL を DB に保存する。 |
| **メタデータの付与** | **同じ登録フォーム** に必要最小限の項目を並べ、送信時にファイル URL とメタデータをまとめて 1 件のレコードとして DB に保存する。 |

## 3. MVP としての推奨構成：Airtable + S3 + Cesium Viewer

- **MVP を安く作る**という目標では、**Airtable + S3 + Cesium Viewer** の組み合わせが、DB 部分の UI 工数を最小にしやすい。
- **役割分担**:
  - **Airtable**: メタデータ・物件の登録・検索の UI（一覧・フィルタ・フォームは Airtable が提供）。CAD が小さい場合は Airtable の添付で保存、大きい場合は「CAD の URL」のみ Airtable に格納。
  - **S3**: 大きな CAD ファイルの実体を保存。自前のアップロード用 API または簡易画面で S3 に上げ、取得した URL を Airtable の該当レコードに記載する。
  - **Cesium Viewer**: 自前でホストする Web アプリ（後述）。検索結果の CAD を表示する。

---

## 4. Cesium Viewer の配信場所（Airtable のサイトに置くか）

- **Cesium Viewer のコードを配信するのは、Airtable のサイトではない。**
- Cesium Viewer は **自前でホストする Web アプリ**（例: Vercel、Netlify、AWS 等）で配信する。ユーザーはその URL にアクセスしてビューアを利用する。
- **Airtable** は「データの入れ物」と「登録・検索の UI」を提供する側。ビューアを動かす HTML/JS を Airtable 上に置く運用は一般的ではない。
- 運用イメージ:
  - **登録・検索・一覧**: Airtable の Base をユーザーに開いてもらう（または Airtable のビューを iframe で自前サイトに埋め込む）。
  - **CAD の表示**: 自前サイトの Cesium Viewer で、Airtable API から取得したレコードの「CAD URL」を読み込み、表示する。

## 5. 自然言語検索と「SQL にするサーバー」・検索結果の表示

### 5.1 自然言語から検索条件への変換

- 「この敷地に合った CAD データを検索して」のような**自然言語**を、そのまま Airtable に投げるのではなく、**どこかで「検索条件」に変換**する必要がある。
- **自前で用意するサーバー（またはサーバーレス関数）** で、次のようにする。
  - ユーザーが Cesium Viewer 側（または検索用 UI）で自然言語を入力する。
  - その文字列を**サーバーに送る**。
  - サーバーで **LLM** を使い、自然言語を解釈して **Airtable 用の検索条件**（`filterByFormula` の式や、クエリパラメータで渡せる条件）に変換する。  
    （Airtable は SQL ではなく REST API と `filterByFormula` 等で検索するため、「SQL にする」のではなく「Airtable のフィルタ式・パラメータにする」形になる。）
  - サーバーが **Airtable API** を呼び出し、条件に合うレコードを取得する。
  - 取得したレコード一覧を**レスポンスとして返す**。Cesium Viewer 側では、その一覧を表形式で表示し、行を選ぶと該当 CAD をビューアで表示する。

### 5.2 検索結果のレコード一覧を表形式で表示する方法

- **方法 A: Airtable の UI をそのまま使う**  
  - 検索条件に合わせて **Airtable のビュー**（フィルタ済みのビュー）をあらかじめ用意しておき、そのビューを **iframe で埋め込む**、または **Airtable の共有リンク** を開いてもらう。  
  - 表形式の一覧は Airtable の画面で表示される。工数は小さいが、「自然言語で出した条件」とビューのフィルタを完全に一致させるには、動的にビューを変えるか、別途「検索結果だけを表示する画面」が必要になる場合がある。

- **方法 B: 自前で表形式の UI を用意する（推奨）**  
  - 上記のサーバーが Airtable API で取得した**レコード一覧を JSON で返す**。  
  - **Cesium Viewer を配信している自前サイト**内に、その一覧を表示する**表（テーブル）**を 1 つ用意する。  
  - 表の各行をクリックすると、そのレコードに紐づく CAD（S3 等の URL）を Cesium Viewer で読み込んで表示する。  
  - 検索結果が「自然言語 → サーバー → Airtable API」の結果と常に一致し、ビューアとの連携（行クリックで表示）も作りやすい。

- **まとめ**: 表形式で見せたいだけなら「Airtable の UI を表示すればよい」という運用（方法 A）も可能。自然言語検索と Cesium 表示をきちんとつなげるなら、**サーバーで Airtable API を叩き、返ってきたレコードを自前の表形式 UI で表示する**（方法 B）が扱いやすい。

## 6. 権限制御・認証の安上がりな定番

- **Airtable 側のアクセス**  
  - Airtable の Base は、**協力者（Collaborator）** や **リンク共有** で開く権限を制御する。有料プランでは「ユーザーごとのアクセス」も設定できる。  
  - API は **Personal Access Token** または **OAuth** で呼び出す。サーバー側でトークンを持ち、**サーバー経由でのみ** Airtable にアクセスするようにすれば、クライアントに Airtable の認証情報を渡さずに済む。

- **Cesium Viewer と自前 API の認証**  
  - **安上がりな定番**としては、**Supabase Auth**（無料枠あり）や **Clerk**（無料枠あり）、**Auth0**（無料枠あり）などで「誰がログインしているか」を管理する。  
  - ユーザーは Cesium Viewer をホストしている自前サイトでログインする。ログイン後の **トークン（JWT 等）** を、自然言語検索や CAD 一覧取得の **自前 API** に付けて送る。  
  - サーバー（またはサーレス関数）でトークンを検証し、**ユーザー ID や組織 ID** に応じて「見てよいレコード」を絞る（例: Airtable のレコードに「組織 ID」を持たせ、API で取得するときにその条件を付ける）。  
  - 課金・プランによる機能制限も、この「認証されたユーザー／組織」をキーに、自前 API と DB（または Airtable のメタデータ）で実装する。

- **S3 の CAD ファイル**  
  - 直 URL をブラウザに渡さず、**署名付き URL** を自前 API で発行し、有効期限を短くする。  
  - 自前 API の認証を通したユーザーだけに署名付き URL を返せば、権限制御と組み合わせられる。
